## 💡 核心创新点

本项目在"人机交互"与"推荐算法"的交叉领域实现了三个维度的创新：

### 1. "表里如一"的双轨情感对齐机制
传统音乐推荐依赖用户行为（点击、收藏），存在严重的"冷启动"和"情绪滞后"问题。
* **外在映射**：通过计算机视觉（CV）实时捕捉用户的面部肌肉变化，量化为 7 种离散情绪（愤怒、厌恶、恐惧、快乐、平静、悲伤、惊讶）。
* **内在共振**：曲库中每首歌均经过 NLP 歌词情感分析（如 `Fearful - 0.92`），将视觉捕捉到的"表"与歌词蕴含的"里"进行精准对齐，实现真正的**"所见即所听"**。

### 2. 毫秒级无感交互
无需填写繁琐的情绪问卷，无需佩戴任何生理传感器。系统以 **Webcam 视频流**作为唯一输入源，将情绪识别的交互成本降至 **0**，实现完全隐式的意图捕获。

### 3. 时间滑窗情绪聚合
人类的情绪是波动的，单帧图像的判断极易因眨眼、侧脸产生噪声。
* 本项目首创性地在后端引入了**时间滑窗机制**（默认 3 秒）。
* 在窗口期内持续累积情绪频次，取**众数作为主导情绪**，极大提升了推荐的稳定性和鲁棒性，拒绝"一帧定论"。

## 🏆 技术优越性

在工程实现层面，本项目采用了多项区别于普通 Demo 的高级架构设计：

### 🔒 前后端物理隔离与视觉穿透
* **痛点**：常规 WebRTC 方案易导致浏览器直接占用摄像头，一旦开启多个标签页或后端需同时处理，必然发生设备冲突。
* **优越性**：本项目采用 **MJPEG 流媒体推流架构**。物理摄像头由 Python 后端独占式管理，前端仅通过 `<img src="/api/video">` 接收编码后的视频帧。既解决了设备独占冲突，又将 CV 推理的算力压力完全保留在服务端，前端做到极致轻量。

### ⚡ 高并发线程安全模型
* 情绪检测运行在独立 Daemon 线程中，Flask 主线程负责 API 响应。
* 通过 `threading.Lock` 与 `threading.Event` 构建**无竞态条件的共享状态池**，确保在 500ms 高频轮询下，情绪数据、窗口计数、视频帧读取绝不会出现脏读或死锁。

### 🛡️ 无缝降级容灾机制
* 前端启动时会自动 Ping 后端 `/api/status`。
* 若后端未启动或模型未加载，前端**自动静默切换至内置 Mock 模拟模式**（内置 50+ 首真实格式英文歌单）。保证在任何网络或环境下，产品演示 **永不失败**。

### 🎨 情绪色谱动态 UI (Chromatic UI)
* 不同于千篇一律的静态界面，前端全局 CSS 变量（`--emotion-color`）与后端情绪数据实时绑定。
* 当检测到"悲伤"时，界面边框、光晕、历史点阵渐变为冷色调蓝；检测到"快乐"时，瞬间切换为暖金色。通过视觉氛围的潜意识暗示，增强产品的沉浸感。
